# Протокол сбора точных данных

## Цель

Собирать по каждому проекту точные данные о плане, факте, объёме и рисках без завышения процентов и без зависающих циклов.

## Общие правила

1. Один проход = один тип данных.
2. Не смешивать план, факт и выводы в одном шаге.
3. Если данных нет, ставить `unknown`.
4. Не считать `100%`, если нет полного набора критериев приёмки.
5. Не делать вывод до завершения сбора всех слоёв.

## Слои данных

Для каждого проекта собирать:

- `plan` - что должно быть сделано.
- `fact` - что реально сделано.
- `evidence` - на чём основан факт.
- `risk` - что мешает завершению.
- `volume` - плановый и фактический объём.
- `progress` - `hologramProgress`, `specProgress`, `progress`.

## Последовательность шагов

### Шаг 1. Реестр проектов

Вход: список доступных проектов.

Действие: определить, какие проекты участвуют в сборе.

Выход: канонический список проектов.

Проверка: в списке нет пропусков и дублей.

### Шаг 2. HOLO/SURFACE

Вход: документы HOLO и SURFACE.

Действие: извлечь stage, инварианты, frozen/fluid/draft-состояния.

Выход: документальный слой плана.

Проверка: для каждого проекта есть явное описание состояния.

### Шаг 3. README и планы

Вход: README, docs/plans, refactoring-plan и аналогичные файлы.

Действие: извлечь явные TODO, текущий фокус и roadmap.

Выход: слой незакрытых задач.

Проверка: каждая крупная незавершённая работа зафиксирована.

### Шаг 4. Кодовая реализация

Вход: основной код проекта.

Действие: определить реально работающие модули, фичи, интеграции.

Выход: слой факта реализации.

Проверка: факт подтверждается кодом, а не только описанием.

### Шаг 5. Тесты и проверки

Вход: тесты, smoke, CI, proof hooks.

Действие: собрать подтверждения работоспособности.

Выход: слой доказательств.

Проверка: есть воспроизводимое подтверждение для ключевых утверждений.

### Шаг 6. Коммиты и свежесть

Вход: история коммитов.

Действие: определить актуальность и динамику работы.

Выход: временной слой.

Проверка: есть понятная последняя точка синхронизации.

### Шаг 7. Противоречия

Вход: все собранные слои.

Действие: найти расхождения между планом, фактом и заявленным статусом.

Выход: список конфликтов и рисков.

Проверка: конфликтов не меньше, чем реально найдено; ничего не домысливается.

### Шаг 8. Расчёт метрик

Вход: plan, fact, evidence, risk.

Действие: посчитать `hologramProgress`, `specProgress`, `progress`.

Выход: финальные проценты и объёмы.

Проверка: `progress` не равен 100 без полного набора критериев.

### Шаг 9. Финальная верификация

Вход: весь собранный пакет.

Действие: проверить, что у каждого проекта есть план, факт, объём, доказательства и риск.

Выход: готовая каноническая запись.

Проверка: нет незаполненных критических полей без `unknown`.

## Критерии качества

- Проценты не завышены.
- План и факт не смешаны.
- Объём измерим.
- Данные воспроизводимы.
- Сбор можно повторить тем же способом без зависания.
