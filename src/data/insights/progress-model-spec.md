# Спецификация обновления прогресса

## Проблема

Текущая модель сводит проекты к одному проценту выполнения и иногда даёт 100% там, где это не соответствует факту.

## Цель

Разделить оценку на два независимых контура:

1. `Голограмма` - оперативный план, то есть текущее состояние реализации и прототипа.
2. `Спека` - водопадный контур, то есть степень закрытия спецификации и полного продукта.

Итоговый процент должен быть производным, а не единственным источником истины.

## Поля

Для каждого проекта фиксируются:

- `hologramProgress` - процент выполнения голограммы.
- `specProgress` - процент выполнения спеки.
- `progress` - итоговый blended score.
- `execution` - отображаемый фактический процент по spec-контурy, если нужен legacy-слой.

## Правила расчёта

1. Приоритет имеют явные значения из каталога проекта.
2. Если явных значений нет, используется мягкий fallback без насыщения до 100.
3. `progress` не должен быть равен 100 только из-за количества булевых отметок в документах.
4. На экранах нужно показывать оба процента, а не один общий.

## UI-правило

- В обзорных блоках показывать `Итог`.
- В детальных блоках показывать отдельно `Голограмма` и `Спека`.
- Не смешивать план и факт в одной подписи.

## Критерии приёмки

- UHA не выглядит как 100% завершённый продукт.
- Genius не выглядит как полностью законченный продукт.
- На страницах есть отдельные значения для планового и водопадного контуров.
- Итоговый процент остаётся вычисляемым и не завышается до 100 без основания.
